# PAI 在 Codex 环境下的迁移与落地：完整工作总结（2026-03-02）

## 一、文档目的与阅读说明

本文是对本项目阶段性工作的完整复盘与系统化总结，目标有三个。第一，沉淀从“Claude 专属体系”到“Codex 可运行体系”的真实工程路径，避免后续团队重复踩坑。第二，以可公开发布的方式，对关键技术决策、边界约束、验证证据、风险处理和取舍理由进行明确说明，便于在 GitHub 开源语境中让新贡献者快速理解项目脉络。第三，把“已经做完什么”“当前还差什么”“下一步优先级是什么”放在同一个文档中，形成可执行、可讨论、可迭代的共享基础。

本文并非宣传文案，而是工程复盘文档。它关注可验证事实，不回避失败与限制，不追求概念上的绝对等价，而强调“在当前工具边界内做到高可用与高可维护”。文中提及的路径、命令、文件位置均来自本仓库实际结构，可直接用于定位。

## 二、项目背景：为什么要做这次迁移

本仓库核心是 PAI（Personal AI Infrastructure）发布体系，历史上主要围绕 Claude Code 的运行机制构建。原体系依赖 `.claude/settings.json`、Hook 生命周期、`CLAUDE.md` 指令路由、状态线与上下文注入等能力，属于“平台深度耦合型”设计。在 Claude 生态中这套机制非常顺滑，但当使用场景转向 Codex 时，立即出现一个现实问题：Codex 并不原生支持 Claude 的 Hook 触发模型，因此无法直接复用原有自动化行为。

用户提出的核心诉求不是“做一个演示级适配”，而是希望：

1. 在 Codex 中能直接用起来。
2. 尽可能接近原体系体验。
3. 以后日常执行 `codex` 时自动进入该模式。
4. 自动附带 `--dangerously-bypass-approvals-and-sandbox` 参数。
5. 尽量还原到接近 100% 的等价能力。

这是一组强约束、强连续性、强体验导向的需求，意味着迁移不能只停留在文档解释层，必须落实到可执行入口、依赖检查、失败语义、跨平台兼容和可验证测试。

## 三、初始状态评估：问题到底在哪里

在动手之前，我们先做了结构评估并确认了几个关键事实。

第一，PAI 的核心资产并不只是提示词文本，而是“文档+技能+Hook+脚本+记忆结构”的组合系统。第二，Claude Hook 机制是自动化主干，很多能力通过生命周期事件串联。第三，Codex 目前缺失等价的内核 Hook 生命周期触发接口。第四，很多 Hook 使用 TypeScript 实现，依赖 Bun 运行时。第五，跨平台执行行为（尤其 Windows）对解释器查找、编码、路径处理非常敏感。

换句话说，真正难点不是“把几份文档复制过来”，而是要在不改动原发布资产的情况下，给 Codex 增加一层可持续的“运行时兼容桥”。

## 四、总体迁移策略：两层桥接而不是一次性硬迁移

经过方案比较，最终采用“静态桥接 + 运行时桥接”的双层架构。

### 4.1 静态桥接层

静态桥接的目标是把 PAI 的核心知识结构（算法、上下文路由、技能分类）映射成 Codex 能读取的项目级指令入口。为此新增了生成工具，自动输出适配版 `AGENTS.md`，让 Codex 会话从一开始就知道该读哪里、按什么顺序读、哪些能力可复用。

这一层解决“认知入口”问题：Codex 不再对 PAI 结构无感，而是拥有明确导航。

### 4.2 运行时桥接层

运行时桥接的目标是尽量模拟 Claude 生命周期事件：`SessionStart`、`PreToolUse`、`PostToolUse`、`Stop`、`SessionEnd`。在 Codex 外部通过 runtime 编排 Hook 执行，使原有行为尽可能延续。

这一层解决“行为自动化”问题：即便没有 Claude 内核 Hook，也能在外部实现可控事件流。

### 4.3 一键入口层

为了让“能跑”变成“好用”，追加了一键启动脚本与环境自检，把复杂命令收敛为短命令调用，同时把依赖状态和故障信息提前暴露。

这一层解决“落地门槛”问题：普通使用无需理解底层细节。

## 五、第一阶段成果：Codex 最小可用桥接

第一阶段围绕“先让 Codex 真正理解 PAI 项目”展开，核心产物是桥接生成器与模板。

### 5.1 关键新增文件

- `Tools/codex_bridge_generator.py`
- `Codex/templates/AGENTS.pai-codex.template.md`
- `Codex/README.md`
- `Releases/codex-v0.1/*`（用于发布分层说明）

### 5.2 设计要点

1. 不改动原 Claude 发布载荷，避免破坏历史版本行为。
2. 通过生成器将路径与版本信息参数化，减少手工维护。
3. 明确写出兼容边界，避免误判“100% 原生”。
4. 优先使用标准库，降低依赖复杂度。

### 5.3 测试与验证

桥接层补齐了单元测试与集成测试，验证点包括：技能目录发现、算法版本读取、模板渲染、临时目录生成与输出校验。配套命令已在本地执行并通过。

这一阶段达成了“可读可用”的第一里程碑，但还未覆盖 Hook 自动化体验。

## 六、第二阶段成果：高等价 Runtime（核心工程增量）

第二阶段是本次工作的技术核心：构建 `Tools/pai_codex_runtime.py`，把事件模型、配置解析、Hook 执行与命令编排统一起来。

### 6.1 Runtime 提供的能力

1. 读取 PAI settings 并解析 hooks。
2. 按事件分发并执行对应 Hook。
3. 支持前后工具调用编排（含阻断语义）。
4. 提供 CLI 子命令供脚本化接入。
5. 支持 `run-bash` 一步完成 pre/post 包装。

### 6.2 关键命令面向

- `session-start`
- `pre-tool`
- `post-tool`
- `stop`
- `session-end`
- `run-bash`

这意味着即便 Codex 本体没有内嵌 Hook，我们也能通过外部 runtime 把生命周期“补齐到可用程度”。

### 6.3 安全与阻断语义

为防止“看似执行成功、实则绕过校验”的风险，`pre-tool` 引入显式阻断规则：

1. Hook 退出码为 2 时阻断。
2. Hook JSON 输出包含 deny/block/blocked 决策时阻断。
3. 关键 Hook（如 `SecurityValidator`、`AgentExecutionGuard`、`SkillGuard`）执行失败或超时时阻断。

这套策略确保安全相关逻辑不因迁移而静默失效。

### 6.4 测试覆盖

runtime 同样补齐了单元与集成测试，覆盖配置解析、事件匹配、命令构建、依赖检查、启动编排等核心路径。相关测试命令已经执行通过，属于“有证据的可运行”。

## 七、第三阶段成果：直接可用（降低操作复杂度）

在有了 runtime 后，仍存在一个实际问题：命令过多、步骤分散、对新用户不友好。第三阶段聚焦“操作体验”。

### 7.1 新增一键入口

- `Codex/runtime/pai-codex.ps1`
- `Codex/runtime/pai-codex.sh`

这两个入口把会话生命周期自动串联，用户无需手动逐条触发事件。

### 7.2 新增环境自检

runtime 增加 `doctor` 命令，用于检查 `codex`、`bun` 等关键依赖可用性，并给出结构化输出，便于自动化诊断。

### 7.3 新增启动编排命令

runtime 增加 `launch-codex`，自动完成：

1. 预处理（含桥接文件生成）。
2. 启动前会话 Hook。
3. 调用 codex 本体。
4. Stop / SessionEnd 收尾 Hook。

至此，从“技术上能运行”升级为“用户可直接执行”。

## 八、依赖治理：Bun 的角色与补齐实践

在实操中，用户提出了关键问题：“bun 是什么，有办法补吗？”这直接指向迁移中最常见断点。

### 8.1 Bun 的作用

本项目多个 Hook 是 TypeScript 脚本，默认依赖 Bun 执行。若无 Bun，相关 Hook 无法运行，尤其可能影响安全相关 pre-tool 检查。

### 8.2 实际处理

已在环境中完成 Bun 安装并验证可用，`doctor` 显示 `bun/codex` 都可用，`missing=[]`。这使 runtime 能执行原有 `.ts` hooks，避免能力缺口。

### 8.3 工程意义

通过 `doctor` + 依赖补齐，项目避免了“脚本存在但运行时缺失”的假可用状态，提升了交付可信度。

## 九、跨平台修复：Windows 兼容是关键落地点

迁移期间对 Windows 侧做了针对性处理，重点包括：

1. 可执行文件路径解析（`codex` 与 `codex.cmd` 的识别与回退）。
2. UTF-8 输出与编码容错，减少日志乱码与解析失败。
3. PowerShell 参数透传行为修复，保证附加参数不丢失。

这部分改造对“看起来小、实际上影响巨大”：很多“偶发失败”都来自平台细节而非业务逻辑。

## 十一、关于“100% 等价”的客观结论

用户曾明确追问“能不能做成 100% 等价”。工程上需要诚实回答：在不改 Codex 内核前提下，无法做到严格的 100% 内核等价，但已达到“高等价可运行”。

### 11.1 无法完全等价的原因

1. Codex 不提供 Claude 同等内核 Hook 接口。
2. Claude 特有状态线/上下文字段无法原封不动复制。
3. Transcript 等内部结构在两端存在差异。

### 11.2 已达成的等价层面

1. 核心文档结构与算法路径可复用。
2. 大多数 Hook 在外部 runtime 下可执行。
3. 会话起止与工具前后事件可编排。
4. 安全关键 pre-tool 具备阻断语义。
5. 一键启动与日常自动接管可用。

### 11.3 已知非阻断噪音

在 `Stop/SessionEnd` 阶段，部分 hooks 因缺少 `transcript_path` 会打印解析错误，但主流程返回成功，不阻断使用。这属于兼容层常见“信息噪音”问题，不是主流程失败。

## 十四、工作方法论：为什么这次迁移比较稳

本次迁移遵循了几个看似朴素但非常有效的工程原则：

1. 先最小可用，再高等价，再低门槛。
2. 不在第一步追求完美等价，先建立可迭代结构。
3. 所有关键结论都要有命令级验证证据。
4. 兼容层必须承认边界，不制造“全部支持”的错觉。
5. 给用户保留逃生路径（`codex-raw`），避免单点失败。

这些原则让项目在面对平台差异时保持节奏，减少了大面积返工。

## 十五、用户体验变化：从“会折腾”到“可日用”

迁移前，想在 Codex 中复用 PAI，用户需要手工理解大量内部结构，且没有统一启动链路。迁移后，体验变化体现在三个层次。

第一层，认知变化：Codex 会话拥有明确入口文件与上下文路由。第二层，行为变化：Hook 生命周期在外部 runtime 下自动编排。第三层，操作变化：`codex` 命令本身被自动接管，用户不必每次输入冗长参数。

这三层叠加后，项目从“技术上可迁移”变成“工作上可持续”。

## 十六、风险清单与现实应对

没有任何迁移是无风险的，关键是识别并处置。

### 16.1 主要风险

1. 依赖缺失导致 Hook 静默失败。
2. 跨平台 shell 行为差异导致参数丢失。
3. Transcript 结构差异导致收尾 Hook 报错噪音。

### 16.2 应对策略

1. 通过 `doctor` 显式暴露依赖状态。
2. 用实测日志确认注入参数是否生效。
3. 把已知噪音列为“非阻断限制”，公开透明。
4. 文档中明确兼容边界并给出替代路径。

## 二十、核心经验（一）：平台迁移首先是“机制映射”

很多迁移失败不是因为实现能力不足，而是因为把问题误判为“代码复制”。本项目的真正挑战是机制映射：如何把 Claude 专有机制映射到 Codex 可执行路径上。双层桥接架构的价值正在于此，它把“不可迁移”拆解成“可映射 + 可替代 + 可降级”的连续空间。

## 二十一、核心经验（二）：安全能力必须前置，不可后补

在工具调用链中，最危险的错误是“默认放行”。所以 pre-tool 阶段必须具备可阻断能力，且关键 Hook 失败应按阻断处理，而不是日志警告后继续执行。这个策略可能会增加误阻断概率，但从安全工程角度看，这个代价远小于静默放行带来的风险。

## 二十二、核心经验（三）：可用性工程决定项目能否被长期使用

即使技术上已经成功迁移，如果日常需要大量手动命令，项目仍会在真实工作流里被放弃。把复杂链路收敛为 `codex` 自动接管，是本次工作里“非炫技但最实用”的部分。它把复杂度从用户侧转移到系统侧，才有可能形成稳定使用习惯。

## 二十三、核心经验（四）：保留原生逃生路径是成熟做法

任何包装层都有失效概率。保留 `codex-raw` 的意义在于：当 wrapper 出现异常时，用户仍能直接进入原生命令，不会因为单层故障导致全链路不可用。这种“双轨并存”策略对生产级工具非常关键。

## 二十四、目前仍建议优化的方向

虽然当前状态已可用，但仍有三类优化值得推进。

### 24.1 降噪优化

为 runtime 增加如 `--quiet-hook-errors` 的开关，把 transcript 类非阻断错误写入日志文件而不是持续刷屏，提升终端可读性。

### 24.2 transcript 兼容增强

在 runtime 注入最小 transcript mock 或字段替代，减少 Stop/SessionEnd 阶段 Hook 因字段缺失而报错。

### 24.3 启动策略强化

可按使用场景添加可选参数策略，例如在非 git 目录场景下是否自动附加 `--skip-git-repo-check`，进一步降低首次使用障碍。

## 二十五、为什么这个阶段已经值得开源发布

判断“是否可以发布”的标准不是“是否完美”，而是“是否可复现、可理解、可维护”。从这三个维度看，本阶段已满足发布条件：

1. 可复现：有明确目录、命令、入口与测试。
2. 可理解：有桥接文档、runtime 文档、任务归档与本总结。
3. 可维护：有边界声明、依赖自检、风险清单和后续路线。

因此，可以作为一个清晰版本对外开放，后续在社区反馈中继续迭代。

## 二十六、面向新贡献者的最短上手路径

建议新贡献者按以下顺序进入项目：

1. 先读 `Codex/README.md` 了解桥接目标。
2. 再读 `Codex/runtime/README.md` 理解事件与命令。
3. 运行 `py Tools/pai_codex_runtime.py doctor` 检查依赖。
4. 执行 `codex --version` 验证自动接管是否生效。
5. 跑测试确认本地环境稳定。

这条路径避免陷入细节海洋，先建立可运行闭环再深入实现。

## 二十七、开源协作建议（治理层）

为避免后续贡献质量波动，建议在 GitHub 侧长期坚持以下协作约束：

1. 所有功能改动必须附验证命令与结果摘要。
2. 涉及 runtime 事件语义变更时必须更新文档。
3. 安全相关 Hook 的降级逻辑必须可审计。
4. 新增命令需说明跨平台差异与回退策略。
5. 大改动使用任务文档进行阶段化记录。

这些约束会增加一点提交成本，但能显著降低维护成本。

## 三十、关于“继续还原 100%”的现实路线图

如果未来继续追求更高等价，建议采用“逐项替换 + 可回归验证”的策略，而不是一次性大改。可按以下顺序推进：

1. 先补 transcript 兼容层，降低收尾错误噪音。
2. 再抽象 Hook 输入适配器，减少脚本对平台字段耦合。
3. 最后再讨论更深层 UI/status 能力替代方案。

每一步都应有回归测试和使用者体验验证，不建议“先重写再看结果”。

## 三十一、发布前检查清单（建议执行）

1. 检查 `doctor` 输出无关键缺失。
2. 验证 `codex --version` 走 runtime 且返回 0。

5. 确认 `codex-raw` 逃生入口可用。

## 三十二、结论

本次工作已经把“Claude 专属体系”成功推进到“Codex 下高等价可用体系”。最关键的成果不是某个单点脚本，而是形成了完整闭环：

- 有入口（桥接文档与 AGENTS 生成）
- 有执行（runtime 生命周期编排）
- 有可用性（一键启动与自动接管）
